# Fix spec: full-codebase review findings (2026-06-09)

Findings from a 5-reviewer pass over all 15 crates (~33K lines). This doc is the
work order for a fix session. Work through it phase by phase, top to bottom.

## How to work this spec

- VERIFY each finding against the current code before fixing it. Line numbers
  drift. Findings marked **[verified]** were re-checked by hand on 2026-06-09;
  everything else came from a reviewer and is plausible but unconfirmed. If a
  finding turns out to be wrong, note that next to it and move on, don't force
  a fix.
- Make the smallest change that fixes the issue. No drive-by refactors.
- After every behavior-affecting phase, run the corpus guard (below) and diff
  against the baseline. Zero-diff is expected for pure perf/dedup work.
  Bug fixes will move counts; eyeball a sample of the delta to confirm each
  changed row is the intended fix, not collateral.
- Run validations SEQUENTIALLY, never two at once (concurrent runs corrupt the
  comparison).
- `cargo clippy --all-targets` must stay at 0 warnings. `cargo test` must pass.
- Add a unit test for each P0/P1 fix where one is cheap (parser cases, skip_root_key
  shapes, comment-prefix parsing are all easy table tests).

## Phase 0: blockers

Corpus-guard delta (2026-06-09): baseline 979 err / 3597 warn; after 979 err / 3597 warn (0-diff). 0.1/0.2 are lock-order fixes (no diagnostic change expected). 0.3 parser fix did not change MD output (mixed-content clause inputs not prevalent in MD). 0.4 converter fix produces correct nested SkipRootKey structure; no diagnostic delta because the old single-level structure was already reaching idea content for validation (categories treated as instances), and idea-type localisation checks were not firing CW225 in the baseline anyway. 0.5 scope subscope direction fix did not change MD output (character->country subscope paths not exercised in the scoped scripts that MD validates).

### 0.1 LSP ABBA deadlock, `ruleset` vs `documents` **[FIXED]**

`crates/lsp/src/main.rs:2868` + `:2895` vs `:1399`/`:1409`.
`revalidate_open_dependents` holds the `ruleset` mutex for the whole sweep and
takes `documents.lock()` inside the loop (the `still_current` check). The
request handlers lock in the opposite order (`documents` then `ruleset`):
hover (1399/1409), completion (~1496/1497), goto_definition (~1601/1602),
references (~1715, ~1770), prepare_rename (~2018), rename (~2068). These are
parking_lot sync locks on tokio workers, so one collision wedges the server.

Fix: establish one lock order everywhere (suggest: `ruleset` before
`documents`), or better, restructure the sweep so it snapshots what it needs
from `documents` before taking `ruleset`, and does the `still_current` check
after dropping `ruleset`. Audit ALL lock pairs in main.rs after fixing, not
just these call sites.

### 0.2 LSP ABBA deadlock, `documents` vs `doc_tokens` **[FIXED]**

`crates/lsp/src/main.rs:2751-2755` vs `:2831-2832`. `debounced_validate` calls
`update_doc_tokens` (takes `doc_tokens.write()`) while holding
`documents.lock()`; `revalidate_open_dependents` takes `doc_tokens.read()` then
`documents.lock()`. Same fix discipline as 0.1, do them together.

### 0.3 Parser interior-quote heuristic swallows clause content **[FIXED]**

`crates/parser/src/parser.rs:253-277`. For `x = { "name" 1 }` the lookahead
after the closing `"` sees `1` (bare content), judges the quote interior, and
the string loop (which only stops at `"`, `\`, `\n`) swallows the `1` AND the
closing `}`. The next statement gets pulled inside `x` and a spurious
"unclosed clause" error fires. Same for `{ "file.wav" volume = 1 }`. Mixed
quoted/unquoted entries on one line are valid game config.

Fix: tighten the "bare content means interior quote" rule. A quote followed by
whitespace then bare content should close the string when the content parses as
a new value/key (digits, identifiers) inside a clause context. The safest shape:
only treat the quote as interior when the bare content is NOT separated from it
by whitespace, or restrict the heuristic to known interior-quote shapes
(`"..." "..."` division-name style on the same line). Add parser tests for:
`x = { "name" 1 }`, `x = { "file.wav" volume = 1 }`, and the two division-name
cases already in the comment at parser.rs:240-241 (must keep passing).

### 0.4 skip_root_key block form is structurally wrong **[FIXED]**

Three coordinated pieces:

- `crates/rules/src/rules_converter/types.rs:228-245`: block form
  `skip_root_key = { A B }` becomes ONE `MultipleKeys([A,B], true)` layer.
  F# (RulesParser.fs:1031-1035) maps each block element to its own NESTED
  layer (`SpecificKey` per element, `any` becomes `AnyKey`).
- `crates/index/src/lib.rs:357-364`: `skip_root_key_matches` treats `any`
  inside `MultipleKeys` as a literal key, despite the comment at
  types.rs:236-238 claiming "wildcard semantics live in the matcher".
- `crates/index/src/lib.rs:456-481` consumes the list as a stack of nested
  levels (`[head, tail @ ..]`), but `crates/validation/src/resolve.rs:13-18`
  (`should_skip_root_key`) does `.any()` over the same list at a SINGLE level,
  and `validate_wrapper_grandchildren` only descends one layer. Any type with
  2+ skip_root_key entries indexes instances at depth N but validates at depth 1.

This is the root cause of the known `skip_root_key = { ideas any }` -> 0 ideas
bug (CW225 flood on idea loc keys). Fix the converter to emit one nested layer
per block element with `any` -> `AnyKey`, keep `MultipleKeys` for the
multi-leaf alternatives form, and make the validator consume the stack the same
way the indexer does (share the descent logic, don't re-implement it). Also
check the minor at types.rs:182,253-267: with multiple skip_root_key leaves,
`should_match` is taken from the last leaf's operator; F# uses the first.

Verify with: an MD run, idea instances must be > 0, and the CW225 idea-loc
flood must drop.

### 0.5 Link input-scope check wrong direction and bypasses the registry **[FIXED]**

`crates/game/src/scope_engine.rs:456-460`:

```rust
|| link.valid_scopes.iter().any(|s| {
    *s == SCOPE_ANY || *s == current || s.as_scope().is_of_scope(current.as_scope())
});
```

Two problems: (a) `is_of_scope` asks "does the REQUIRED scope satisfy the
current scope", which is backwards; (b) it round-trips through the hardcoded
`as_scope()` table instead of `self.registry.is_subscope_or_eq`, so
config-driven subscope relations (HOI4 `character is_subscope_of country`)
never apply to link traversal. Country-valid links used from character scope
produce WrongScope/CW245 false positives.

Fix: `self.registry.is_subscope_or_eq(current, *s)`. Also check the dotted-chain
path (`resolve_single` around :364) uses the same predicate. Watch the hardcoded
302/301 special case nearby; dynamically assigned registry ids could collide
with it.

### 1.1 Cross-kind rule matches silently accepted **[verified, leaf side]** **[FIXED]**

`crates/validation/src/rule_core.rs:248-264` and `:303-336`.
`validate_leaf_against_rule`'s NodeRule arm does nothing when the leaf value is
not a Clause (no else-error), and `validate_node_against_rule` does nothing for
a LeafRule candidate. `matching_candidates` includes both kinds
(rule_core.rs:887-909) and `pick_best_candidate` (:342-365) takes the first
candidate with zero errors, so any key with both a leaf and a node rule can
never report a value error: the wrong-kind candidate "validates cleanly" and
wins. Fix: emit a kind-mismatch error (or mark the candidate non-matching) when
a NodeRule meets a non-Clause leaf or a LeafRule meets a node. Expect this to
surface new (real) errors on MD; sample them.

### 1.2 Key-side enum matching diverges from `enum_contains` **[FIXED]**

`crates/validation/src/rule_core.rs:1104-1109` (and `:1032-1037` in
`alias_pattern_matches`): case-sensitive `v == key`, no empty-enum permissive
handling, no `@`-constant handling. The value-side `enum_contains`
(common.rs:219-251) does all three. Share `enum_contains` for key-side checks.
This is the same FP class as the fixed `containerOrientations` case.

### 1.3 Scope diagnostics anchor at 0,0 for node-form usage **[FIXED]**

`crates/validation/src/rule_core.rs:1231-1233, 1275-1277, 1297-1303`. All scope
diagnostics in `validate_alias_usage` use `leaf.map(...).unwrap_or((0, 0))`;
when the usage came through `validate_node_against_rule`, `leaf` is None and
CW248/CW104/105/106/CW267 report at line 0. Thread the node position in (the
call sites have it). The `leaf.map` at :1297-1299 inside the else of
`if let Some(leaf)` is provably always None, dead expression.

Related: `validate_with_type` hardcodes `block_pos = (0,0)` (rule_core.rs:46-48,
:169), so required-field errors on empty-bodied entities anchor at 0,0. Thread
the entity node position.

### 1.4 Modifier key lookup is case-sensitive **[FIXED]**

`crates/validation/src/rule_core.rs:536` and `:610`:
`modifier_keys.map(|mk| mk.contains(&key))` against the as-authored key, while
the surrounding key handling is case-insensitive. Build the set lowercased in
`build_modifier_keys` (loc_field.rs:18-39) and look up lowercased.

### 1.5 Int range check truncates i64 to i32 **[FIXED]**

`crates/validation/src/rule_core.rs:1655-1657`: `let v_i = *v as i32;` wraps.
`research_cost = 4294967296` wraps to 0 and passes. Compare in i64.

### 1.6 SpecificField narrowing and value matching case bugs **[FIXED]**

- `:385-394`: the "specific rule wins over catch-all" filter compares
  `s == key` case-sensitively while the matcher used `eq_ignore_ascii_case`.
- `:1749-1756`: value-side SpecificField is case-sensitive and has an Int arm
  but no Float arm (rule literal `0.5` vs `Value::Float(0.5)` falls to false).
- subtype.rs:93-95, 107-109 + common.rs:160-181: subtype discriminator key
  match (`|s| s == *k`) and `child_key_matches` are case-sensitive and don't
  unquote, while `validate_children` lowercases and unquotes. A mismatched-case
  discriminator silently picks the wrong subtype.

Normalize all of these to the same lowercase+unquote convention.

## Phase 2: major correctness, rules converter and scope registry

Corpus-guard delta: same run as Phase 1 (changes applied together). Phase 2 items produced no additional diagnostic count changes on the MD corpus (HOI4 rules do not exercise CSV loc, and the scope/directive fixes affect edge cases not present in MD). The +158/-76 delta reported under Phase 1 is the combined Phase 1+2 total.

### 2.1 Comment directives match substrings across ALL comment tiers **[FIXED]**

`crates/rules/src/rules_converter/comment_directives.rs:30,63,73,89,104`: every
option except `scope` is detected by bare `contains(...)` over all accumulated
comments, including `#` plain comments and `###` doc lines. A doc line saying
"cardinality" or a commented-out `# cardinality = 0..1` becomes the rule's
option. Same class at types.rs:300 (type_key_filter), types.rs:330
(graph_related_types), types.rs:365-367 (required/optional/primary), and
subtypes.rs:137-146. Also `find` takes the OLDEST match while
`parse_required_scopes` scans `rev()` for the newest.

Fix: parse the prefix level FIRST (exactly `##`, not `#` and not `###`), then
match `key = value` at line start, newest-match-wins, in ONE shared helper that
all directive extraction goes through. This is the `##` vs `#` bug class that
parse_required_scopes was already fixed for, applied everywhere.

### 2.2 parse_required_scopes regressed the prefix distinction **[FIXED]**

`comment_directives.rs:243-247`: `trim_start_matches('#')` strips ALL leading
hashes, so `# scope = X` and `### scope = the owning country` both register as
a `## scope` option. Require exactly `##`. Folds into the 2.1 helper.

### 2.3 Unresolvable link scopes silently dropped **[FIXED]**

`crates/validation/src/scope.rs:73-78` + `crates/game/src/scope_registry.rs:51-57`:
link `input_scopes`/`output_scope` names that fail `id_of` are dropped via
`filter_map`. A dropped input scope NARROWS the link (WrongScope FPs); an
unresolvable output_scope demotes a scope-change link to value-only. Also
`id_of` doesn't map `"none"` (F# maps none -> anyScope). Fix: map `none`,
and log + treat unresolved names as ANY rather than dropping.

### 2.4 Loc command scope-chain threading is broken **[FIXED]**

`crates/localization/src/scope_validation.rs:225-262` +
`crates/localization/src/yaml_parser.rs:278-303` +
`crates/localization/src/loc_string.rs:168-171`. `parse_bracket` routes EVERY
dotted command to `JominiCommand`, the parser flattens all bracket chains into
one Vec (chain boundaries lost), and `validate_jomini_chain` checks each
segment against a fresh context at initial scope. So `[overlord.owner.GetName]`
checks `owner` from the initial scope: false CW260s and missed real ones.
Meanwhile the correct dot-split loop in `validate_command_string` (:152-217) is
dead in production; its tests hand-craft inputs the real parser never produces.

Fix: keep chain boundaries in the parsed representation (Vec per bracket, not
one flattened Vec), and thread scope through segments the way
`validate_command_string` does. Delete or rewire the dead path so there is one
scope-threading implementation.

### 2.5 Indexer-vs-validator type matching drift **[FIXED]**

`crates/index/src/lib.rs:309-349` (`check_path_dir`) ignores `path_file` and
`path_extension`; the validator (resolve.rs:138-152) treats both as hard
filters. Files get indexed under types they don't validate under: phantom
instances, duplicate-name pollution. Fix by sharing one path-match predicate
between index and validation (same direction as the skip_root_key fix in 0.4;
ARCHITECTURE_BUGS.md already flags this fork as "highest correctness-stakes
dup").

### 2.6 CSV loc files fed to the YAML parser **[FIXED]**

`crates/localization/src/service.rs:139-150`: `walk_folder` collects `.csv`
(CK2/VIC2) but everything goes to `parse_loc_text`, which fails on CSV. Every
CSV file becomes a PARSE ERROR and its keys never join the union (mass false
CW225). `csv_parser.rs` exists with zero production callers. Route `.csv`
through it. Also :145-148: a failed read becomes `(String::new(), None)`,
surfacing as a misleading "empty file" error; propagate the IO error.

### 2.7 Variable flatten loop is a no-op, deleted vars persist **[FIXED]**

`crates/info/src/lib.rs:492-499` + `:601-608`: the "flatten non-@-var entries
back into the legacy map" loop keeps only names that START with `@` (the items
already there), so it's a no-op and the comment is inverted. Consequence:
`clear_file` only removes @-vars, so `value_set` variable names are never
removed from `all_variables`; deleted variables persist in completion all
session. Invert the condition; add a test that defines a value_set var, clears
the file, and asserts it's gone.
